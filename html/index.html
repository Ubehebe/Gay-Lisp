Gay Lisp

The first standards-compliant Scheme that runs in the browser

What is Lisp?
One of the oldest and most influential programming languages. Kind of like Latin is to European languages.
What is Scheme?
A modern dialect of Lisp, renowned for its beauty but seldom used in the real world. Kind of like Italian.
What is Gay Lisp?
A Scheme interpreter conforming to R5RS that runs 100% in the browser, no plugins or downloads required. You can also run it on a server.
What do I need to run Gay Lisp?
If you are reading this, Gay Lisp is probably running on your computer. Gay Lisp is written in an unfancy version of JavaScript that has been included in browsers for more than a decade, so it should work even in non-contemporary browsers. Of course, it will not work if you have disabled JavaScript.
This website, however, uses some more recent bells and whistles, so an up-to-date browser is recommended.
How does it work?
There is a regex-based scanner and a recursive descent parser, both handwritten. Programs are incrementally transformed into continuation-passing style and evaluated on a trampoline. Primitive Scheme values are implemented by their primitive JavaScript counterparts as much as practical; in particular, the semantics of numerical operations is mostly that of JavaScript, while still complying with the Scheme standard. All the non-primitive syntax and procedures are written in Scheme and loaded at boot time.
Is the phrase “the first standards-compliant Scheme that runs in the browser” justified?
I believe so. Most JavaScript Scheme implementations provide a useful subset of Scheme without aiming for compliance with any standard. Typically, this means standard features that significantly complicate the implementation are omittted:
  • BiwaScheme lacks macros.
  • Fargo has incomplete library support (no dynamic-wind, call-with-values, etc.)
  • jsScheme does not support nested ellipses in macro transformers. This means the implementation cannot provide the canonical definition of do.
By contrast, Gay Lisp aims for complete compliance with R5RS. You can jump between the specification and the implementation in order to test features. And you can run every example in the specification straight from the specification.
Please open an issue on GitHub if you know of a more complete JavaScript Scheme implementation.
Why are you targeting R5RS instead of the newer R6RS standard?
As of early 2012, the status of R6RS as a community standard is in doubt. There is talk of splitting standarization efforts into two tracks, one of which could abrogate features introduced by R6RS.
Is this useful for anything?
There is some experimental DOM interoperability:
